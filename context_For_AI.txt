# PROMPT DE DIRECIONAMENTO PARA IA

Você está continuando o desenvolvimento de um **jogo Dungeon Crawler 3D top-down baseado em grid** desenvolvido em **Unity + C#**.

Antes de gerar qualquer código:

1. Considere que **GridManager é a fonte de verdade do estado do mundo**.
2. **Nunca use `transform.position` como fonte lógica de posição.**
3. Toda a entidade deve possuir **`Vector2Int gridPosition`**.
4. Sempre que mover uma entidade, **atualize primeiro o grid e depois o transform**.
5. Sempre respeite os sistemas já existentes antes de criar novos.
6. Prefira **estender sistemas existentes** ao invés de criar novos sistemas paralelos.
7. Não introduza dependências externas ou frameworks novos sem necessidade.
8. Mantenha código **simples, modular e legível**.

Antes de modificar qualquer script, considere:

* Como o sistema atual funciona
* Se a mudança pode quebrar o grid
* Se a mudança pode quebrar a IA
* Se a mudança pode deixar tiles ocupados incorretamente

Se um objeto pode morrer ou desaparecer, **ele deve sempre liberar seu tile no grid**.

# 1. VISÃO GERAL DO PROJETO

Tipo de jogo:

**Dungeon Crawler 3D Top-Down baseado em grid**

Arquitetura central:

```text
Grid lógico (2D)
↓
Representação visual (3D)
Toda lógica ocorre no grid.
O mundo 3D é apenas visualização.

2. PRINCÍPIO FUNDAMENTAL DO PROJETO
GRID É A FONTE DA VERDADE
Tudo depende do GridManager.
O grid controla:

ocupação de tiles

colisão (lógica, sem colliders físicos para entidades)

posição de entidades

pathfinding

visão de inimigos

Estrutura principal:

C#
TileType[,] tiles
Tipos de tile atuais:

C#
Empty
Wall
Player
Enemy
Item
Water
Lava
3. REGRA CRÍTICA DE MOVIMENTO
Toda a movimentação deve seguir exatamente esta ordem:

Plaintext
1 atualizar grid (tile antigo → Empty)
2 atualizar gridPosition
3 atualizar grid (tile novo → tipo da entidade)
4 atualizar posição no mundo
Exemplo correto:

C#
GridManager.Instance.SetTile(oldX, oldY, TileType.Empty);
gridPosition = newPosition;
GridManager.Instance.SetTile(gridPosition.x, gridPosition.y, TileType.Enemy);
UpdateWorldPosition();
Nunca faça:

C#
transform.position = ...
sem atualizar o grid primeiro.

4. CONVERSÃO GRID ↔ MUNDO
Sempre usar funções do GridManager.

C#
GridToWorld(x,y)
WorldToGrid(position)
Exemplo:

Plaintext
Grid (10,5) → World (10,0,5)
Eixo Y é altura.

5. SISTEMAS PRINCIPAIS DO JOGO
GridManager
Responsável por:

armazenar grid

validar tiles

conversões de posição

posição do player

Nunca duplicar lógica de grid fora dele.

PlayerController
Responsável por:

input de movimento

movimentação restrita à grelha

Combate Automático: Ao esbarrar num inimigo, entra em "Engage" e ataca automaticamente por intervalos de tempo. Recuar para um tile vazio aciona o "Disengage".

Busca de alvos através de pura correspondência de gridPosition (sem Physics.OverlapSphere).

EnemyController
Responsável por:

Movimentação via IA

Pathfinding A* nativo (contorna paredes)

Gestão Separada de Combate/Movimento: Possui cronómetros separados para moveInterval e attackInterval.

CombatStats & DamageData
CombatStats: Define atributos base (força, minDamage, maxDamage) e gera um objeto DamageData.

DamageData: Pacote de dados imutável com cálculos de mitigação, críticos e falhas.

Health
Controla a vitalidade e a UI das entidades de forma auto-suficiente.
Funções principais:

Instanciação dinâmica da sua própria UI (healthBarPrefab).

Feedback visual autônomo através de instanciação ("fire and forget") do damageSplashPrefab.

Trancamento de HP em 0 para evitar bugs de UI.

Limpeza do Grid antes de Destruir: Executa o Die() garantindo que o TileType.Empty é aplicado à posição morta.

6. REGRA CRÍTICA DE MORTE
Quando uma entidade morre:

liberar tile

destruir objeto

Isso evita tiles bloqueados permanentemente.

7. IA DOS INIMIGOS
A IA possui três pilares que correm de forma independente da UI:

1 Visão
Baseada na distância de Manhattan até ao jogador.

2 Pathfinding
Algoritmo utilizado:

*A (A-Star)**

Heurística:

Plaintext
Manhattan Distance
Movimentos permitidos (4 direções, sem diagonais).
Implementa limite de iterações (maxIterations) para evitar quebra de framerate.

3 Comportamento
Estados:

Idle: Movimento aleatório.

Chase: Perseguição via A*.

Combat: Paragem da movimentação e ativação do attackTimer automático (corpo a corpo).

8. SISTEMA DE VIDA, UI e EFEITOS
Health (Script): Lida com o dano lógico. Instancia os prefabs visuais.

HealthBar (Script): Apenas ouve o Health. Está blindado matematicamente com Mathf.Clamp01.

SimpleSplash (Script): Elemento visual que flutua e se autodestrói (orientado para a câmera isométrica).

Regra: A lógica do jogo NUNCA deve parar se houver um erro visual na UI ou Efeitos.

9. CÂMERA
Script:

Plaintext
CameraFollow
Características Atuais:

Segue o transform do player através de Vector3.SmoothDamp.

Grid Bounds: Possui limites definidos para não mostrar áreas vazias além das extremidades do Grid.

Anti-Clipping: Utiliza Physics.Linecast contra a camada "Wall" para afastar a câmera caso bata no cenário.

Offset padrão: (0,20,-15).
